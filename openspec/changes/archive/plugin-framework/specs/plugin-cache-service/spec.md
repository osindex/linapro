## Requirements

### Requirement: Dynamic Plugins Access Distributed Cache via Named Spaces

The system SHALL provide cache service via MySQL MEMORY table with namespace isolation and strict length validation.

#### Scenario: Plugin accesses authorized cache space
- **WHEN** a plugin calls cache service get/set/delete/incr/expire
- **THEN** the host only allows access to authorized `host-cache` resources
- **AND** data is stored in the shared MEMORY table

#### Scenario: Oversized value is rejected with explicit error
- **WHEN** a plugin writes data exceeding namespace/key/value length limits
- **THEN** the host returns an explicit error without truncating or partial write

#### Scenario: Unauthorized cache space is rejected
- **WHEN** a plugin calls an unauthorized cache space
- **THEN** the host rejects the call

### Requirement: Source Plugins Must Access Host KV Cache via Plugin-Scoped Cache Facade

The system SHALL provide a governed cache facade through source plugin `HostServices` service directory. Source plugins can only access cache through plugin-visible `namespace`, logical `key`, and TTL; they must not receive the host internal `kvcache.Service`, `OwnerType`, encoded owner key, coordination KV, Redis client, SQL table backend, or provider.

#### Scenario: Source plugin obtains cache service via HostServices
- **WHEN** a source plugin accesses `HostServices().Cache()` in HTTP registrar, Cron registrar, or hook payload
- **THEN** the system returns the current plugin-scoped bound cache service
- **AND** the service only accepts plugin-visible `namespace`, logical `key`, cache value, and TTL parameters
- **AND** the service does not expose internal cache backend, owner type, or underlying client

#### Scenario: Source plugin cache service missing
- **WHEN** the source plugin call path has not configured the cache service
- **THEN** the system must not temporarily create a new host cache service graph in the call path
- **AND** the caller must receive a clear error or nil service explicitly handled by plugin code

### Requirement: Source Plugin Cache Key Must Be Generated by Host by Plugin and Tenant Scope

The system SHALL generate internal cache keys inside the source plugin cache facade based on the current `pluginID`, `namespace`, logical `key`, and current tenant context. Source plugins MUST NOT pass or override `pluginID`, owner key, owner type, or tenant key.

#### Scenario: Different source plugins isolated under same namespace
- **WHEN** source plugins `plugin-a` and `plugin-b` both write to `namespace=profile` and `key=current`
- **THEN** the system generates different internal cache keys for the two plugins
- **AND** `plugin-a` cannot read `plugin-b`'s cached value
- **AND** `plugin-b` cannot read `plugin-a`'s cached value

#### Scenario: Write source plugin cache under current tenant context
- **WHEN** a source plugin writes cache in the request context of tenant `1001`
- **THEN** the system generates an internal cache key containing tenant `1001`, plugin ID, namespace, and logical key
- **AND** other tenant contexts reading the same plugin, same namespace, and same logical key must not hit that tenant's cache

#### Scenario: Write source plugin cache without tenant context
- **WHEN** a source plugin writes cache in a startup period, platform-level task, or test call without tenant context
- **THEN** the system generates a platform-level plugin cache key
- **AND** the key must still include plugin ID, namespace, and logical key

### Requirement: Source Plugin Cache Must Reuse Host Startup-Injected Shared Cache Service

The system SHALL inject the shared `kvCacheSvc` created during HTTP startup into the source plugin cache facade. The source plugin cache facade MUST reuse the shared instance or its shared backend and must not call `kvcache.New()` to create an independent cache service graph in plugin registration, request processing, hook callbacks, cron callbacks, or cache method call paths.

#### Scenario: Standalone mode source plugin cache uses standalone backend
- **WHEN** `cluster.enabled=false` and a source plugin calls cache `set`
- **THEN** the source plugin cache facade executes the write through the startup-injected shared `kvCacheSvc`
- **AND** the system may use SQL table backend or host standalone cache strategy
- **AND** it does not require the coordination KV backend to exist

#### Scenario: Cluster mode source plugin cache uses coordination KV backend
- **WHEN** `cluster.enabled=true` and a source plugin calls cache `set`
- **THEN** the source plugin cache facade executes the write through the startup-injected shared `kvCacheSvc`
- **AND** the shared service uses the coordination KV backend behind the host unified coordination provider
- **AND** the source plugin cache facade does not independently resolve Redis configuration or create a Redis client

### Requirement: Source Plugin Cache Operations Must Maintain Lossy Cache and TTL Semantics

The system SHALL treat source plugin cache as lossy cache. Source plugin cache MUST NOT be used as the source of truth for permissions, configuration, plugin stable state, tenant isolation, authoritative business data, critical cache revision numbers, or cross-instance consistency coordination. Source plugin cache TTL MUST express `time.Duration` semantics; negative TTL must return a clear error.

#### Scenario: Source plugin reads non-existent or expired cache
- **WHEN** a source plugin reads a non-existent or expired cache key
- **THEN** the system returns a cache miss
- **AND** the system must not require the caller to treat the cache value as authoritative business state

#### Scenario: Source plugin sets cache with TTL
- **WHEN** a source plugin writes a cache value and passes a positive TTL
- **THEN** the system sets the expiration time in a backend-agnostic TTL semantic
- **AND** standalone SQL table backend handles expiration through existing expiration check and cleanup tasks
- **AND** cluster coordination KV backend handles expiration through backend-native TTL

#### Scenario: Source plugin passes negative TTL
- **WHEN** a source plugin calls `set`, `incr`, or `expire` with a negative TTL
- **THEN** the system returns a clear error
- **AND** the system must not write or modify the corresponding cache value

### Requirement: Source Plugin Cache Write Failure Must Not Disguise as Success

The system SHALL return errors when source plugin cache write, delete, increment, or expire operations fail. The system MUST NOT report success to the source plugin when the shared cache backend, coordination KV, SQL table, or key validation fails.

#### Scenario: Source plugin cache write backend failure
- **WHEN** a source plugin calls `set`
- **AND** the shared cache backend returns a connection, validation, or persistence error
- **THEN** the source plugin cache facade returns an error
- **AND** the system must not return a success write cache snapshot to the plugin

#### Scenario: Source plugin increments string cache value
- **WHEN** a source plugin executes `incr` on an existing string cache value
- **THEN** the system returns a structured type error
- **AND** the original string value must not be modified
