## 背景

插件管理页面首次进入时会调用 `GET /api/v1/plugins` 加载完整治理列表。该接口当前在请求路径中执行插件清单扫描、动态 WASM artifact 解析、注册表快照读取、运行时状态投影、依赖检查、宿主服务表注释解析和 cron 声明补充。冷路径集中在首个列表请求上，导致第一次打开页面明显慢。

用户认可优化方向为：复用请求内重复扫描结果、在宿主启动后异步预热完整插件管理读模型、在插件生命周期和动态产物变化后显式失效缓存。暂不做列表瘦身，因为列表字段被详情、安装、卸载、升级等弹窗复用，拆分会把时延转移到弹窗打开阶段，影响用户体验。

## 变更内容

- 让插件列表构建在同一次请求中复用已扫描的 manifest 和依赖 snapshot，避免每个插件依赖检查重复触发清单扫描。
- 新增插件管理列表读模型缓存，缓存内容保持与现有列表响应字段一致，不删减弹窗依赖的治理信息。
- 宿主启动完成后异步预热插件管理列表读模型；预热失败不阻塞服务启动，首个请求仍可回退到现有同步构建路径。
- 插件同步、上传、安装、卸载、启用、禁用、源码升级、动态升级和租户供应策略更新后显式失效读模型，并复用现有插件运行时缓存修订机制在集群模式下触发跨实例失效。

## 影响

本变更影响插件管理列表查询性能和插件治理缓存一致性，不改变公开 API 路径、HTTP 方法、响应字段、前端列表字段或弹窗信息结构。不新增数据库表，不新增前端文案，不改变数据权限边界。
