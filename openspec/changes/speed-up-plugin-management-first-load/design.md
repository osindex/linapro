## 目标

降低插件管理页面首次进入时的后端接口耗时，同时保留完整列表字段，确保详情、安装、卸载、升级和授权确认弹窗仍可直接复用列表返回的治理信息。

## 方案

1. 请求内复用清单扫描结果
   - `ReadOnlyList` 已经扫描出完整 manifest 列表，应将该列表写入请求上下文。
   - 依赖检查构建 dependency snapshot 时优先读取请求上下文中的 manifest 列表，避免每个插件再次 `ScanManifests`。
   - 依赖检查现有的请求级 snapshot cache 继续保留，用于复用 registry/release 派生结果。

2. 完整列表读模型缓存
   - 在 plugin facade 内维护进程内插件管理列表读模型缓存，缓存 `ListOutput` 的完整投影。
   - `List` 在无过滤条件或有过滤条件时都先读取完整缓存，再做内存过滤，响应字段不瘦身。
   - 缓存未就绪、构建失败或被失效后，请求同步构建并回填缓存，保证功能可用。

3. 启动异步预热
   - HTTP runtime 完成插件启动一致性校验、生命周期挂载和 cron 启动后，后台异步调用插件服务预热列表读模型。
   - 预热失败只记录日志，不阻塞 HTTP 服务启动。
   - 预热必须使用启动根上下文派生的 detached context，避免请求取消影响后台任务。

4. 缓存失效与集群一致性
   - 权威数据源：插件 manifest、动态 WASM artifact、`sys_plugin`、`sys_plugin_release`、`sys_plugin_resource_ref`、插件菜单/权限投影、插件 cron 声明和 host service 声明。
   - 一致性模型：单机模式使用进程内缓存和本地显式失效；集群模式复用 `pluginruntimecache` 的共享修订号，读路径先 `EnsureFresh` 后读缓存。
   - 失效触发点：插件显式同步、动态包上传、安装、卸载、启用、禁用、源码升级、动态升级、动态运行时协调结果、租户供应策略更新。
   - 最大可接受陈旧时间：单机为同进程生命周期操作后立即失效；集群为现有插件运行时修订号刷新预算。
   - 故障降级：缓存构建失败时返回同步构建错误；启动预热失败不阻塞服务，后续请求继续尝试构建；集群修订读取失败时沿用现有保守失败策略。

## 边界

- 不拆分列表字段，不新增详情弹窗懒加载接口，不删除 `DependencyCheck`、host service、route review、mock data、cron 声明等既有列表投影信息。
- 不改变插件管理 API 的 REST 契约。
- 不新增数据库结构。
- 不新增或修改 i18n 资源。
- 不新增数据操作接口，列表查询仍保持无副作用。

## 影响评估

- i18n：不新增前端文案、菜单、路由、接口文档字段或翻译资源；缓存失效会复用现有插件 i18n 运行时 bundle 失效逻辑，不扩大语言资源治理范围。
- 数据权限：插件管理列表仍是平台治理接口，不新增业务数据查询、详情读取、下载、聚合统计或插件数据访问路径。
- 缓存一致性：本变更新增插件管理列表读模型缓存，按单机/集群显式区分策略。`cluster.enabled=false` 时使用进程内缓存和本地失效；`cluster.enabled=true` 时复用插件运行时共享修订号触发跨实例失效，读路径在消费缓存前执行 freshness 检查。
- 开发工具/脚本：不修改开发工具、构建脚本、测试脚本或 CI 辅助入口。
