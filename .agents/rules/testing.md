# 测试与验证规则

## 适用范围

本规则约束单元测试、E2E 测试、反馈修复测试覆盖、测试 helper、fixture、页面对象、runner 配置、治理验证和任务完成前的验证门禁。

## 强制遵守场景

以下任一场景命中时，必须先读取并遵守本规则：

- 新增或修改单元测试、E2E 测试、测试 helper、fixture、页面对象、runner 配置或测试数据
- 修复 bug、处理反馈、改变功能行为或用户可观察结果
- 修改权限、缓存、数据权限、插件桥接、接口联动或端到端流程
- 纯治理类反馈需要选择 OpenSpec 校验、静态扫描、文件检查或格式检查作为验证方式

## 单元测试要求

- 单元测试必须自包含且顺序无关。
- 每个测试方法必须自行构造和清理所需场景与数据。
- 禁止依赖其他测试方法先运行或遗留的副作用。
- 需要共享逻辑时只能共享 helper/fixture，并由当前测试显式调用。
- 清理逻辑必须使用当前测试注册的`defer`、`t.Cleanup`或等价机制完成。
- 涉及全局状态时，当前测试必须保存原值并恢复。

## E2E 测试要求

- 测试用例必须完整覆盖业务模块的各项操作，例如增删改查，保证功能完整性和可用性。
- 所有 E2E 用例需要在`tasks.md`中有工作记录，并且使用`lina-e2e`技能生成和管理对应测试用例。
- 修复 bug 或新增功能涉及用户可观察行为变化时，必须编写或更新对应 E2E 测试用例。
- 涉及用户可观察页面、路由、表单、表格、权限交互、接口联动或端到端工作流的 bugfix，必须使用 E2E 测试。
- 纯内部重构且无 UI 变化可豁免新增 E2E，但需运行已有测试套件或更贴合的验证确认无回归。
- 使用 Playwright 等测试工具创建截图时，文件必须放置到项目根目录`temp/`目录下。

## E2E 文件组织要求

- E2E 测试文件名必须使用模块本地递增的`TC{NNN}-{brief-name}.ts`前缀。
- 同一模块目录内编号从`TC001`开始连续递增。
- 拒绝旧的全局递增编号、四位`TC{NNNN}`文件名、目录内重复编号或因其他模块编号产生的跳号。
- 源码插件专属 E2E 必须保留在插件自己的`hack/tests/e2e/`目录。
- 插件专属 POM/helper 必须保留在同插件的`hack/tests/pages/`和`hack/tests/support/`目录，不得回流到宿主测试目录。

## Bugfix 反馈测试要求

- 所有以`lina-feedback`或等价反馈流程处理、且涉及功能行为、业务逻辑、用户可观察结果、运行时接口、缓存、权限、数据权限、插件桥接等可执行行为变化的 bugfix，必须新增或更新能够复现问题并验证修复有效性的自动化测试。
- 后端纯逻辑、服务层、工具函数、缓存、权限、数据权限、插件桥接等内部行为优先使用单元测试。
- 用户可观察行为或跨模块工作流缺陷必须使用 E2E 测试。
- 测试应覆盖原始问题的失败场景和修复后的预期行为。
- 功能行为类 bugfix 缺少单元测试或 E2E 测试不得标记完成，也不得通过`lina-review`。

## 治理类反馈验证要求

- 纯项目治理类反馈不要求新增单元测试或 E2E。
- 适用验证方式包括`openspec validate`、静态扫描、文件存在性检查、格式检查或审查结论。
- 治理类反馈必须说明为什么不需要运行时测试，并提供匹配的治理验证证据。

## 任务完成门禁

- 修改完成后必须运行相关新增或更新测试、识别出的回归测试或对应治理验证，并确认通过，再标记任务完成。
- 如果回归测试失败且与当前变更相关，必须直接修复。
- 如果发现独立问题，应记录为新的反馈任务。

## 审查要求

- 审查必须判断变更是否涉及可执行行为变化。
- 审查必须确认功能行为类 bugfix 具备单元测试或 E2E 覆盖。
- 审查必须确认治理类反馈具备匹配治理验证证据。
- 审查必须确认测试可单独运行、顺序无关且清理完整。
- 审查必须确认 E2E 文件编号和插件测试目录归属合规。
